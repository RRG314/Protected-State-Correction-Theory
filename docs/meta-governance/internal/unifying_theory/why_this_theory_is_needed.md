# Why This Theory Is Needed

Date: 2026-04-17

## 1) Problem Without the Framework

Without a unifying framework, the ecosystem splits into isolated claims:
- exact projector theorems,
- anti-classifier no-go theorems,
- bounded-domain obstruction stories,
- restricted nonlinear identifiability studies,
- domain-specific closure/obstruction results.

This fragmentation creates two failure modes:
1. false universal claims,
2. missed shared structure across branches.

## 2) What the Framework Adds

The framework adds a precise center:
- universal recoverability core (factorization/collision),
- branch-limited theorem strengthening,
- strict separation of theorem, validated evidence, engineering consequence, and analogy.

## 3) Examples Showing Need

### Example E1 (amount-only heuristic failure)
Equal rank or equal budget can still produce opposite exactness verdicts on supported restricted classes.
Implication: decision rules based on amount-only summaries are unsafe without structural compatibility checks.

### Example E2 (family-enlargement trap)
Exactness on a narrow family can fail immediately on enlarged family.
Implication: “works on my family” is not a stable theorem claim unless enlargement fragility is addressed.

### Example E3 (bounded-domain trap)
Divergence reduction can succeed while bounded protected-class exactness fails due to boundary compatibility.
Implication: scalar residual success is not sufficient certification.

### Example E4 (nonlinear restricted lane)
In restricted soliton lane, equal observation count can have opposite identifiability outcomes.
Implication: structure-over-amount theme persists beyond linear branch in restricted form.

### Example E5 (MHD domain split)
Variable-resistivity lane admits annular survivors while smooth axis-touching nonconstant survivors fail in supported classes.
Implication: domain/regularity class is structural, not cosmetic.

## 4) Counterexamples Showing Why Weaker Heuristics Fail

1. Rank-only classifier counterexample family (`OCP-049`).
2. Fixed-budget-only classifier counterexample family (`OCP-050`).
3. Bounded-domain transplant counterexample (`OCP-023`/`OCP-028`).
4. Model-mismatch decoder error-floor witness (`OCP-053`).
5. Soliton same-count opposite-verdict witness package (supported family).

## 5) Practical Consequence

The framework is needed because it gives one disciplined way to move from:
- naive amount-based reasoning,
- to structural compatibility diagnostics,
- to explicit branch-limited theorem decisions and safe nonclaims.
